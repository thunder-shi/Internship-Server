# Internship 部署执行步骤（唯一文档）

**按本文从前往后操作，即可完成首次部署与日常发版。**  
本文与 `**docker-compose.yml`**、`**.env.example`**、`**docker/nginx/gateway.conf**` 配套，均位于 `**server_directory/**` 内。

---

## 一、`server_directory` 是什么、和代码仓库的关系


| 说明         | 内容                                                                                                                  |
| ---------- | ------------------------------------------------------------------------------------------------------------------- |
| **用途**     | 仅用于 **Docker Compose 部署**（编排、入口 Nginx、环境变量模板）。                                                                      |
| **不要做的事**  | 勿将 `**server_directory` 整体** push 进 **Internship-Front** 或 **Internship-NewBack**（避免部署文件混入业务仓库）；可单独 zip 备份或自建小仓库管理。 |
| `**.env`** | 只在服务器本目录生成，**禁止**提交到任何 Git。                                                                                         |


**本机（开发电脑）建议目录：**

```text
Internship/
├── server_directory/          ← 本文件夹：整体上传到服务器
│   ├── docker-compose.yml
│   ├── docker/nginx/gateway.conf
│   ├── .env.example
│   ├── .gitignore
│   └── 部署执行步骤.md        ← 本文
├── Internship-Front/          ← 前端 Gitee，单独 clone / push
└── Internship-NewBack/        ← 后端 Gitee，单独 clone / push
```

**服务器上（示例 `/opt/internship`）必须与上结构一致**：`server_directory` 与两个代码目录**同级**；**所有 `docker compose` 命令在 `server_directory` 内执行**。

`docker-compose.yml` 中 `build.context` 为 `**../Internship-Front`**、`**../Internship-NewBack`**，依赖上述相对位置，**勿随意改目录名**。

## 二、架构说明（便于理解排错）

### 2.1 本栈包含的 7 个服务


| 服务           | 镜像 / 构建                           | 说明                                          |
| ------------ | --------------------------------- | ------------------------------------------- |
| `mysql`      | `mysql:8.0`                       | 业务库，数据在 Docker 卷 `internship_mysql_data`    |
| `redis`      | `redis:7-alpine`                  | 缓存 / 会话等                                    |
| `minio`      | `minio/minio:RELEASE…`（见 compose） | 对象存储；**不映射宿主机端口**，仅栈内访问                     |
| `kkfileview` | `keking/kkfileview:4.4.0`         | 在线预览；经入口 Nginx 的 `**/kkfileview/`**         |
| `newback`    | 构建 `../Internship-NewBack`        | Spring Boot，`SPRING_PROFILES_ACTIVE=docker` |
| `front`      | 构建 `../Internship-Front`          | 静态资源（容器内 Nginx）                             |
| `nginx`      | `nginx:1.26-alpine`               | 统一入口：反代 `/api/`、`/kkfileview/`、其余到 `front`  |


### 2.2 浏览器访问路径

- 对外：`**http://<服务器IP>:<HTTP_PUBLISH>`**（默认端口 `**18088**`，见 `.env.example`）。  
- 前端请求 `**/api**` → 入口 Nginx → `**newback:8111**`。  
- 文件预览 → `**/kkfileview/**` → kkFileView。

### 2.3 后端配置与 `application.properties`

容器内 MySQL / Redis / MinIO 连接由 `**docker-compose.yml` 中的环境变量** 提供（如 `SPRING_DATASOURCE_URL`、`SPRING_DATA_REDIS_*`、`MINIO_*`），**会覆盖** `Internship-NewBack` 里 `application.properties` 的同类项。  
因此：**按本文 Docker 部署时，不必为迁服务器去改 `application.properties`**；本地 IDE 仍可用原文件连旧环境。

## 三、与 1Panel / OpenResty 共存（端口与反代）

- 若宿主机 **80 已被 OpenResty（1Panel）占用**：在 `**.env`** 中使用 `**HTTP_PUBLISH=18088`**（或 `ss -tlnp` 确认过的其它空闲端口），浏览器访问 `**http://IP:18088**`。  
- **想用 80、地址栏不写端口**：在 **1Panel / OpenResty** 中为域名配置 **反向代理**，目标 `**http://127.0.0.1:18088`**（端口与 `.env` 中 `HTTP_PUBLISH` 一致）。  
- 本栈 **MinIO 不映射 9000/9001**，与机器上其它 MinIO **不冲突**；防火墙一般只需放行 `**HTTP_PUBLISH`** 与 **SSH 22**。

## 四、Redis 与 Spring Profile


| 场景                       | 做法                                                                                                                                                                                                   |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Docker 本栈（生产）**        | Compose 已设 `**SPRING_PROFILES_ACTIVE=docker`**；`application-docker.properties` 会**覆盖**默认里对 Redis 自动配置的 `exclude`，使容器内能连 Redis。**勿**在容器里再叠加会排除 Redis 的配置。                                             |
| **本地无 Redis、IDE 直接运行后端** | 默认 `application.properties` 仍**排除** `RedisAutoConfiguration`。可选：① 本机起 Redis 与配置地址一致；② 临时注释该 `spring.autoconfigure.exclude` 行（勿将注释状态提交仓库）；③ 在 IDE 的 VM options 中自行设置与 Spring Boot 版本兼容的 exclude 覆盖方式。 |

## 第七步（可选）：从旧环境迁移 MySQL

> 全新项目可跳过。库名与 `**.env` 中 `MYSQL_DATABASE`** 一致。

### 7.1 导出（在能连旧库的环境）

```bash
mysqldump -h <旧库IP> -P 3306 -u<用户> -p --single-transaction --routines --triggers --default-character-set=utf8mb4 <旧库名> > internship.sql
```

将 `**internship.sql**` 上传到 `**/opt/internship/server_directory/**`。

### 7.2 查 MySQL 容器名

```bash
cd /opt/internship/server_directory
docker compose ps --format json | head
docker ps --format "{{.Names}}" | grep -i mysql
```

记下 **mysql** 容器名，下文称为 `**MYSQL_CONTAINER`**。

### 7.3 导入

在 `**server_directory`** 下（`internship.sql` 与本目录同级）：

```bash
# 将 MYSQL_CONTAINER 换成实际容器名；密码与 .env 中 MYSQL_ROOT_PASSWORD 一致
docker exec -i MYSQL_CONTAINER mysql -uroot -p'你的MYSQL_ROOT_PASSWORD' internship < internship.sql
```

若 `internship` 库尚未创建，可先：

```bash
docker exec -i MYSQL_CONTAINER mysql -uroot -p'你的MYSQL_ROOT_PASSWORD' -e "CREATE DATABASE IF NOT EXISTS internship CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```

再执行导入。

**MySQL 5.7 → 8.0 常见报错**：若出现 `Variable 'sql_mode' can't be set to the value of 'NO_AUTO_CREATE_USER'`，说明旧 dump 含 5.7 专用项。在 `server_directory` 下备份并处理后重导：

```bash
cp internship.sql internship.sql.bak
sed -i "s/NO_AUTO_CREATE_USER,//g" internship.sql
sed -i "s/,NO_AUTO_CREATE_USER//g" internship.sql
sed -i "s/NO_AUTO_CREATE_USER//g" internship.sql
```

若导入在中途失败，建议清空库后整库重导：

```bash
docker exec -i MYSQL_CONTAINER mysql -uroot -p'你的MYSQL_ROOT_PASSWORD' -e "DROP DATABASE IF EXISTS internship; CREATE DATABASE internship CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
docker exec -i MYSQL_CONTAINER mysql -uroot -p'你的MYSQL_ROOT_PASSWORD' internship < internship.sql
```

### 7.4 验证

```bash
docker exec -it MYSQL_CONTAINER mysql -uroot -p -e "USE internship; SHOW TABLES;"
```

## 第八步：从旧环境迁移 MinIO（必做）

> **从旧生产环境迁数据时，本步必做**（附件、上传文件在 MinIO 中）。  
> **kkFileView** 无业务数据需迁，新容器即可。  
> 前置：已完成 **第一～七步**（栈已 `up`，`.env` 中 `MINIO_`* 已配置）。

### 8.1 对照信息（本项目固定配置）

**旧环境**（`Internship-NewBack` → `application.properties`，**本地 IDE 用，迁移时不要改**）：


| 项                     | 值                           |
| --------------------- | --------------------------- |
| 地址                    | `http://47.96.172.199:9000` |
| AccessKey / SecretKey | `minioAdmin` / `minioAdmin` |
| 桶名                    | `internship`                |


**新栈**（`server_directory/.env` + `docker-compose.yml` 中 `newback` 环境变量）：


| 项               | 值                                 | 说明                                                                                                    |
| --------------- | --------------------------------- | ----------------------------------------------------------------------------------------------------- |
| 容器内 API（后端实际使用） | `http://minio:9000`               | compose 中 `MINIO_ENDPOINT`                                                                            |
| 迁移时宿主机访问新 MinIO | `http://127.0.0.1:19000`          | 仅 **8.2** 临时 `ports` 映射，供 `mc` 使用                                                                     |
| 账号 / 密码         | `minioadmin` / `internship_minio` | `.env` 的 `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD`，对应 compose 的 `MINIO_ACCESSKEY` / `MINIO_SECRETKEY` |
| 桶名              | `internship`                      | `.env` 的 `MINIO_BUCKET`，对应 compose 的 `MINIO_BUCKETNAME`                                               |


`.env` 示例（勿提交 Git）：

```env
MINIO_ROOT_USER=minioadmin
MINIO_ROOT_PASSWORD=internship_minio
MINIO_BUCKET=internship
```

> 迁完后在浏览器 **:18088** 新上传的文件，只写入 **Docker 卷 `internship_minio_data`**（容器 `internship-minio-1`），**不会**再写到 `47.96.172.199:9000`。

### 8.2 为新 MinIO 临时映射本机端口（仅迁移期间）

编辑 `docker-compose.yml`，在 `**minio`** 服务的 `volumes` 与 `networks` 之间增加（**只绑 127.0.0.1**，勿对公网暴露）：

```yaml
    ports:
      - "127.0.0.1:19000:9000"
```

应用并确认监听：

```bash
cd /opt/internship/server_directory
docker compose up -d minio
ss -lntp | grep 19000
```

### 8.3 安装 MinIO Client（`mc`）

在 **ecm-ddc4** 上执行（若无 `mc`）：

```bash
wget -O /usr/local/bin/mc https://dl.min.io/client/mc/release/linux-amd64/mc
chmod +x /usr/local/bin/mc
mc --version
```

### 8.4 配置别名并同步数据

在 **ecm-ddc4** 上执行（**8.2** 已加临时 `ports` 且 `docker compose up -d minio` 之后）。  
**旧端** 与 `application.properties` 一致；**新端** 与 `.env` + `docker-compose.yml` 中 `minio` / `newback` 一致。

```bash
cd /opt/internship/server_directory

# 旧 MinIO（application.properties：47.96.172.199:9000，minioAdmin / minioAdmin）
mc alias set oldminio http://47.96.172.199:9000 minioAdmin minioAdmin

# 新 Docker MinIO（compose 临时映射 127.0.0.1:19000 → 容器 9000；.env：minioadmin / internship_minio）
mc alias set newminio http://127.0.0.1:19000 minioadmin internship_minio

# 确认旧桶 internship 存在
mc ls oldminio/internship

# 新桶（与 MINIO_BUCKET、minio.bucketName 一致）
mc mb newminio/internship 2>/dev/null || true

# 全量同步：旧桶 internship → 新桶 internship
mc mirror oldminio/internship newminio/internship

# 同步结果
mc ls newminio/internship
mc du newminio/internship
```

说明：

- 旧 AK 为 `**minioAdmin**`（注意大小写），新栈为 `**minioadmin**`（`.env` 全小写），二者不同属正常。  
- `mc mirror` 对象多时会较久；若中断可再次执行同一命令（会增量补齐）。  
- 若 `mc alias set oldminio` 失败，先确认旧机 **9000** 仍可达且旧 MinIO 进程在跑。

### 8.5 删除临时端口并恢复 compose

迁移完成后，**必须**去掉临时映射，避免长期暴露本机 9000 代理。

1. 编辑 `docker-compose.yml`，将 `**minio`** 恢复为**仅卷 + 网络**（删除下面两行，保留其余不变）：

```yaml
    ports:
      - "127.0.0.1:19000:9000"
```

删除后 `minio` 段应类似：

```yaml
  minio:
    image: minio/minio:RELEASE.2024-11-07T00-52-20Z
    restart: unless-stopped
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: ${MINIO_ROOT_USER}
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD}
    volumes:
      - internship_minio_data:/data
    networks:
      - internship_net
```

1. 重新应用并重启后端（使连接仍走栈内 `minio:9000`）：

```bash
cd /opt/internship/server_directory
docker compose up -d
docker compose restart newback
```

1. 确认 **19000 已不再监听**（可选）：

```bash
ss -lntp | grep 19000
# 无输出即正常
```

日常：`**newback**` 通过 compose 环境变量 `MINIO_ENDPOINT=http://minio:9000`、`MINIO_ACCESSKEY=minioadmin`、`MINIO_SECRETKEY=internship_minio`、`MINIO_BUCKETNAME=internship` 访问新 MinIO，**不经过** `application.properties` 中的 `47.96.172.199`。

### 8.6 验证

**容器与配置**

```bash
cd /opt/internship/server_directory
docker compose ps
docker exec internship-newback-1 printenv | grep -E '^MINIO_'
```

应看到类似：

```text
MINIO_ENDPOINT=http://minio:9000
MINIO_ACCESSKEY=minioadmin
MINIO_SECRETKEY=internship_minio
MINIO_BUCKETNAME=internship
```

**新 MinIO 桶内对象**（可选；**8.4** 已执行 `mc du` 可跳过。若 **8.5** 已删临时 `ports`，用栈内网络检查）：

```bash
docker network ls | grep internship
# 网络名一般为 internship_internship_net，以下按实际名称替换 NETWORK
docker run --rm --network internship_internship_net --entrypoint sh minio/mc -c \
  'mc alias set local http://minio:9000 minioadmin internship_minio && mc ls local/internship | head'
```

**业务验证（必做）**

1. 浏览器打开 `http://<服务器IP>:18088`（与 `.env` 中 `HTTP_PUBLISH` 一致），登录。
2. 测试 **上传新文件** → 应写入 **Docker MinIO** 桶 `internship`。
3. 测试 **下载 / kkFileView 预览** 旧附件 → 依赖 **8.4** 的 `mc mirror` 是否完整。

**失败时**

```bash
docker compose logs -f --tail=100 newback
```

核对 `.env` 与上表 `MINIO_*` 是否一致；勿在容器内误用 `application.properties` 的 `47.96.172.199:9000`。

### 8.7 常见问题


| 现象                          | 处理                                                                         |
| --------------------------- | -------------------------------------------------------------------------- |
| `mc alias set oldminio` 连不上 | 旧 MinIO 是否仍在跑；IP/端口、AK/SK 是否正确；安全组/防火墙是否放行旧机 **9000**                      |
| 旧桶名不是 `internship`          | `mc ls oldminio` 查看后，修改 `mc mirror oldminio/<旧桶名> newminio/<MINIO_BUCKET>` |
| 新密码含特殊字符                    | `mc alias set` 中密码用 **单引号** 包裹                                             |
| 误删临时端口前未 mirror 完           | 重新加上 `ports` 并 `docker compose up -d minio` 后重跑 `mc mirror`                |

## 第十步：日常发版（代码更新后）

> **适用**：首次部署（第一～九步）已完成，线上已在跑 Docker 栈。  
> **目的**：把 **Internship-Front** / **Internship-NewBack** 仓库的新提交发布到服务器，**不丢**已有 MySQL / MinIO / Redis 数据。

### 10.1 发版会影响什么、不会影响什么


| 操作                                          | 是否执行             | 数据是否保留                                       |
| ------------------------------------------- | ---------------- | -------------------------------------------- |
| `git pull` + `docker compose up -d --build` | 日常发版             | **MySQL、MinIO、Redis 卷保留**；用户在网站上的增删改、已上传文件仍在 |
| 重新导入 `internship.sql`                       | 仅换库 / 灾难恢复时      | 会覆盖或冲掉当前库内容，**非日常发版**                        |
| 重新 `mc mirror` MinIO                        | 仅换桶 / 灾难恢复时      | 一般 **非日常发版**                                 |
| `docker compose down -v`                    | **禁止**（除非有意清空环境） | **会删除所有命名卷，数据丢失**                            |


发版时 **会重建** 的通常是 `**front`、`newback` 镜像**；`mysql`、`redis`、`minio` 等 **有数据卷的服务** 一般 **只重启或不重建**，业务数据在卷里。

### 10.2 本机（开发电脑）：提交并推到远程

在各自仓库根目录操作（**不要**把 `server_directory` push 进 Front / NewBack）。

**后端 `Internship-NewBack`：**

```bash
cd Internship-NewBack
git fetch origin
git pull --rebase origin main
# 本地改代码、自测
git status
git add <要提交的文件>
git commit -m "feat: 说明本次改动"
git push origin main
```

**前端 `Internship-Front`：**

```bash
cd Internship-Front
git fetch origin
git pull --rebase origin main
# 本地 npm install / npm run build 自测（可选）
git status
git add <要提交的文件>
git commit -m "fix: 说明本次改动"
git push origin main
```

说明：

- 分支若不是 `main`，将上述 `main` 改为实际分支名。  
- 推送前若 `git push` 被拒绝，先 `git pull --rebase origin main` 再 `git push`。  
- **勿**在容器用的 Docker 配置未进仓库时只在服务器手改代码（下次 `git pull` 会丢）。

### 10.3 服务器：拉代码并构建发布（标准流程）

SSH 登录 **ecm-ddc4**（或你的部署机），**按顺序**执行：

```bash
# 1. 后端
cd /opt/internship/Internship-NewBack
git fetch origin
git pull --rebase origin main

# 2. 前端
cd /opt/internship/Internship-Front
git fetch origin
git pull --rebase origin main

# 3. 构建并滚动更新（必须在 server_directory 执行）
cd /opt/internship/server_directory
docker compose up -d --build
```

`--build` 会根据代码变更重新构建 `front` / `newback` 镜像并替换容器；**无需**再执行第七步 MySQL、第八步 MinIO。

### 10.4 只更新了一侧时的做法


| 变更范围                                               | 服务器最少操作                                                                                                                     |
| -------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **仅后端**                                            | `Internship-NewBack` 里 `git pull` → `server_directory` 里 `docker compose up -d --build`（Compose 会检测变更，主要重建 `newback`）       |
| **仅前端**                                            | `Internship-Front` 里 `git pull` → 同上（主要重建 `front`）                                                                          |
| **前后端都有**                                          | 两个仓库都 `git pull` → 再 `compose up -d --build`                                                                                |
| **仅 `server_directory`**（compose、nginx、`.env`、本文档） | 本机 WinSCP / scp 更新该目录 → **不要** `git pull` Front/Back → `cd server_directory && docker compose up -d`（改动了镜像构建相关时再 `--build`） |


### 10.5 发版后验证

```bash
cd /opt/internship/server_directory
docker compose ps
```

确认 **7 个服务** 均为 **Up**（`mysql`、`redis` 为 **healthy** 更佳）。

```bash
docker compose logs -f --tail=80 newback
```

浏览器访问（端口以 `.env` 中 `HTTP_PUBLISH` 为准，示例 **18088**）：

- `http://<服务器IP>:18088/`  
- 登录后抽查：**列表增删改**、**文件上传**、**附件预览**。

### 10.6 发版后数据在哪里（答疑）

在 **本套 Docker 站点**（如 `:18088`）上的操作：

- **增删改** → 容器 `**internship-mysql-1`**，卷 `**internship_mysql_data**`  
- **新上传文件** → 容器 `**internship-minio-1`**，卷 `**internship_minio_data**`，桶 `**internship**`（与 `.env` 中 `MINIO_BUCKET` 一致）

发版 **不会** 把数据写回旧环境 `47.96.172.199:9000` 或旧 MySQL；只要 **不删卷**、访问的是新站点，数据都找得到。

### 10.7 常见问题


| 现象                        | 处理                                                                                                   |
| ------------------------- | ---------------------------------------------------------------------------------------------------- |
| `git pull` 冲突             | 在对应仓库解决冲突后 `git add` → `git rebase --continue` 或按团队规范 merge，再 `compose up -d --build`                |
| `docker compose build` 失败 | 看报错：前端 `npm run build`、后端 `mvn package`；查磁盘 `df -h`、Docker Hub 是否超时                                  |
| 页面仍是旧版                    | 浏览器 **强刷**（Ctrl+F5）；确认 `front` 镜像时间 `docker images internship-front`                                 |
| 接口 502                    | `docker compose logs newback`；`mysql` / `redis` 是否 healthy                                           |
| 想回滚某一版                    | 在仓库 `git checkout <旧commit>` 或 `git pull` 到旧 tag，再 `docker compose up -d --build`（**勿**随意 `down -v`） |


### 10.8 一键命令速记（前后端都更新时）

**推荐：自动发版脚本（拉 GitHub `main` + 构建，写日志；需已配置 `scripts/auto-deploy.sh` 中的 Token）**

```bash
bash /opt/internship/server_directory/scripts/auto-deploy.sh
```

查看日志：`tail -50 /var/log/internship-deploy.log`

**或逐条手动执行（与脚本逻辑相同）：**

```bash
cd /opt/internship/Internship-NewBack && git fetch origin && git pull --rebase origin main
cd /opt/internship/Internship-Front && git fetch origin && git pull --rebase origin main
cd /opt/internship/server_directory && docker compose up -d --build
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
```

> 若要 **换库 / 导入 `internship.sql`**，另见 **第七步**；日常发版 **不要** 在脚本或上述命令中执行 SQL 导入。

### 10.9 定时自动发版（cron）

适用于开发人员频繁 push 到 GitHub `main`、服务器需每日自动拉代码并构建的场景。

**脚本位置**（在 `server_directory` 内，随本目录一并上传到服务器）：


| 文件                            | 说明                                                          |
| ----------------------------- | ----------------------------------------------------------- |
| `scripts/auto-deploy.sh`      | 拉 NewBack / Front 的 `main` + `docker compose up -d --build` |
| `cron/internship-deploy.cron` | 北京时间 **1:00** 执行；**2:00** 仅当 1:00 失败时重试                     |


**不会在定时任务中执行**：`internship.sql` 导入、MinIO 迁移、`server_directory` 从 Git 更新。

**服务器首次安装：**

```bash
# 1. 编辑脚本，填入 GitHub Personal Access Token（勿提交到 Git）
nano /opt/internship/server_directory/scripts/auto-deploy.sh
# 修改 GITHUB_TOKEN="..."

# 2. 可执行权限
chmod +x /opt/internship/server_directory/scripts/auto-deploy.sh

# 3. 安装 cron（需 root）
cp /opt/internship/server_directory/cron/internship-deploy.cron /etc/cron.d/internship-deploy
chmod 644 /etc/cron.d/internship-deploy

# 4. 手动试跑（若报 `No such file or directory`，见下方说明）
bash /opt/internship/server_directory/scripts/auto-deploy.sh
```

若执行脚本报 `: No such file or directory`：

1. 确认文件已上传：`ls -la /opt/internship/server_directory/scripts/auto-deploy.sh`
2. 多为 Windows 换行符导致，在服务器执行：`sed -i 's/\r$//' /opt/internship/server_directory/scripts/auto-deploy.sh`
3. 或用 `bash` 显式调用（见上），并 `chmod +x` 脚本

**日志与状态：**

- 运行日志：`/var/log/internship-deploy.log`
- 最近一次结果：`server_directory/logs/deploy-last.status`（`SUCCESS` / `FAIL` + 时间戳）

**超时（脚本内可调）：** `git fetch` / `git pull` 各 120s；`docker compose up -d --build` 600s。

## 第十一步：常用运维命令

```bash
cd /opt/internship/server_directory

docker compose ps
docker compose logs -f --tail=200 newback
docker compose restart newback
docker compose stop          # 停止，卷保留
docker compose down          # 删容器，默认卷仍保留
```

## 附录 D：故障排查


| 现象         | 可检查项                                                         |
| ---------- | ------------------------------------------------------------ |
| 浏览器打不开     | 安全组 / firewalld；`HTTP_PUBLISH`；`docker compose ps` 中 `nginx` |
| 502 / 接口失败 | `docker compose logs newback`；MySQL 是否 healthy               |
| 登录异常       | Redis 密码与 `.env`；确认未在 Docker 中误加排除 Redis 的 profile/参数        |
| 预览失败       | kkFileView 日志；MinIO presigned；trust 配置                       |
| 构建失败       | 服务器访问 Docker Hub / Maven Central；磁盘空间                        |
